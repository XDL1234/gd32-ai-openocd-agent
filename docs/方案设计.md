# GD32 AI 自动化开发 Agent 方案

## 一、项目定位

构建一个基于 Claude Code 的 GD32 自动化开发 Agent，通过 Skills + Bash 调用 OpenOCD，实现：

1. 自动读取硬件文档和工程源码
2. 生成工程分析文档
3. 制定任务计划
4. 执行编译、烧录、串口观察、调试

**核心原则**：AI 负责理解和规划，Bash 负责确定性执行。

---

## 二、整体架构

```
用户 GD32 工程
    ↓
硬件文档 / 源码 / 配置文件
    ↓
Claude Code
    ↓
内置 Skills
    - gd32-openocd    (编译、烧录、调试)
    - document-skills (文档读取)
    - hardware-analysis (硬件分析)
    ↓
Bash 直接调用
    - CMake / Make / GCC
    - OpenOCD
    - GDB
    - 串口工具
    ↓
GD32 硬件板卡
```

**简化点**：去掉复杂 CLI，直接用 Skills + Bash 调用底层工具。

## 五、AI 自动识别芯片型号

AI 读取工程文件，自动识别芯片型号，并与硬件文档对比验证。

### 识别来源

| 来源 | 文件 | 识别内容 |
|------|------|----------|
| 启动文件 | `startup_*.s` | 芯片系列（GD32F10x、GD32F30x 等） |
| 链接脚本 | `*.ld` | Flash/SRAM 大小、起始地址 |
| 头文件 | `gd32f10x.h`、`gd32f30x.h` | 芯片系列宏定义 |
| CMakeLists.txt | `CMakeLists.txt` | 芯片型号宏（GD32F103C8T6） |
| Makefile | `Makefile` | 芯片型号宏 |
| 源码 | `main.c` | 头文件引用 |

### 识别规则

```python
# 识别优先级
1. 启动文件名 → 芯片系列
   - startup_gd32f10x_md.s → GD32F10x Medium
   - startup_gd32f30x_hd.s → GD32F30x High

2. 链接脚本 → Flash/SRAM
   - GD32F103C8Tx_FLASH.ld → 64KB Flash, 20KB SRAM

3. 头文件 → 芯片系列
   - #include "gd32f10x.h" → GD32F10x 系列

4. CMakeLists.txt → 芯片型号
   - -DGD32F103C8T6 → 具体型号

5. 源码头文件引用 → 芯片系列
   - #include "gd32f10x_libopt.h" → GD32F10x
```

### 识别流程

```bash
# 1. 扫描启动文件
find . -name "startup_*.s" | head -5

# 2. 扫描链接脚本
find . -name "*.ld" | head -5

# 3. 扫描头文件
find . -name "gd32*.h" | head -5

# 4. 扫描 CMakeLists.txt
grep -r "GD32" CMakeLists.txt 2>/dev/null

# 5. 扫描 Makefile
grep -r "GD32" Makefile 2>/dev/null

# 6. 扫描源码头文件引用
grep -r "#include.*gd32" src/ inc/ 2>/dev/null
```

## 七、Skills 设计

### 1. gd32-openocd Skill

核心 Skill，负责编译、烧录、调试：

```markdown
---
name: gd32-openocd
description: GD32 编译、烧录、调试。当需要编译工程、烧录固件、串口观察、寄存器调试时使用。
allowed-tools: Read, Bash, Write, Glob, Grep
---

# GD32 OpenOCD Skill

## 编译工程
检查编译系统类型，执行对应命令：
- CMake: mkdir -p build && cd build && cmake .. && make
- Make: make clean && make

## 烧录固件
使用 OpenOCD 烧录：
openocd -f .gd32-agent/openocd.cfg -c "program build/app.elf verify reset exit"

## 串口观察
使用串口工具读取输出：
- Windows: python -m serial.tools.miniterm COMx 115200
- Linux: minicom -D /dev/ttyUSBx -b 115200

## 寄存器调试
使用 GDB + OpenOCD：
openocd -f .gd32-agent/openocd.cfg &
arm-none-eabi-gdb build/app.elf -ex "target remote :3333"

## 安全规则
- 烧录前必须确认芯片型号和调试器
- 禁止自动擦除或解锁
- 编译失败禁止烧录
```

## 八、自动化流程

### Phase 0：环境初始化

用户输入：`初始化这个 GD32 工程`

Claude 执行：
```bash
# 创建目录结构
mkdir -p hardware workflow docs/{imported,analysis,tasks,reviews} .gd32-agent

# 生成模板文件
cp templates/hardware.md hardware/hardware.md
cp templates/development-flow.md workflow/development-flow.md

# 生成 OpenOCD 配置模板
cat > .gd32-agent/openocd.cfg << 'EOF'
source [find interface/stlink.cfg]
source [find target/gd32f1x.cfg]
adapter speed 1000
EOF
```

### Phase 1：读取硬件文档

```bash
# 读取硬件文档
cat hardware/hardware.md

# 提取关键信息
# - MCU 型号
# - 调试器类型
# - 串口配置
# - 风险限制
```

### Phase 2：工程扫描

```bash
# 扫描工程结构
find . -name "*.c" -o -name "*.h" -o -name "CMakeLists.txt" -o -name "Makefile"

# 读取关键文件
cat CMakeLists.txt
cat src/main.c
cat startup_*.s
cat *.ld
```

### Phase 3：生成分析文档

生成 `docs/analysis/project-hardware-analysis.md`：

```markdown
# 工程与硬件分析

## 硬件信息
- MCU：GD32F103C8T6
- 调试器：ST-LINK
- 串口：USART0 / 115200

## 源码扫描
- 工程类型：CMake / Make
- 启动文件：startup_gd32f10x_md.s
- 链接脚本：GD32F103C8Tx_FLASH.ld

## 一致性检查
| 项目 | 硬件文档 | 源码 | 结果 |
|------|----------|------|------|
| 芯片系列 | GD32F10x | GD32F10x | 一致 |
```

### Phase 4：任务规划

用户输入需求后，生成：
- `docs/tasks/task-requirements.md`
- `docs/tasks/task-plan.md`

使用 Plan Mode 制定执行计划，用户确认后再执行。

### Phase 5：代码修改

根据 task-plan.md 执行代码修改，更新 `docs/tasks/task-progress.md`。

### Phase 6：编译

```bash
# CMake 工程
mkdir -p build && cd build && cmake .. && make 2>&1 | tee ../.gd32-agent/build.log

# Make 工程
make clean && make 2>&1 | tee .gd32-agent/build.log
```

### Phase 7：烧录

```bash
# 烧录前确认
echo "芯片：$(grep '芯片型号' hardware/hardware.md | cut -d'：' -f2)"
echo "调试器：$(grep 'LINK 类型' hardware/hardware.md | cut -d'：' -f2)"
echo "固件：build/app.elf"

# 执行烧录
openocd -f .gd32-agent/openocd.cfg -c "program build/app.elf verify reset exit"
```

### Phase 8：串口观察

```bash
# Windows
python -m serial.tools.miniterm COM3 115200 --raw > .gd32-agent/serial.log &
sleep 10
kill %1

# Linux
timeout 10 minicom -D /dev/ttyUSB0 -b 115200 -C .gd32-agent/serial.log
```

### Phase 9：寄存器调试

```bash
# 启动 OpenOCD
openocd -f .gd32-agent/openocd.cfg &
OPENOCD_PID=$!

# 使用 GDB 读取寄存器
arm-none-eabi-gdb build/app.elf -batch \
  -ex "target remote :3333" \
  -ex "monitor halt" \
  -ex "info registers" \
  -ex "monitor reg" \
  > .gd32-agent/register-dump.md

# 停止 OpenOCD
kill $OPENOCD_PID
```

## 九、OpenOCD 配置

### 自动生成配置

根据 hardware.md 生成 `.gd32-agent/openocd.cfg`：

```python
# generate_openocd_config.py
def generate_config(hardware_md):
    # 解析硬件文档
    probe_type = parse_probe(hardware_md)  # stlink / daplink / jlink
    chip_series = parse_chip(hardware_md)  # gd32f1x / gd32f3x / gd32f4x

    # 生成配置
    config = f"""
source [find interface/{probe_type}.cfg]
source [find target/{chip_series}.cfg]
adapter speed 1000
"""
    return config
```

### 支持的调试器

| 调试器 | OpenOCD 配置 |
|--------|--------------|
| ST-LINK | interface/stlink.cfg |
| DAPLink | interface/cmsis-dap.cfg |
| J-Link | interface/jlink.cfg |

### 支持的 GD32 系列

| 系列 | OpenOCD 配置 |
|------|--------------|
| GD32F1x | target/gd32f1x.cfg |
| GD32F3x | target/gd32f3x.cfg |
| GD32F4x | target/gd32f4x.cfg |

## 十四、总结

**核心理念**：硬件文档定事实，流程文档定规矩，Skills 负责能力，Bash 负责执行。

**简化策略**：去掉复杂 CLI，直接用 Skills + Bash 调用底层工具，降低实现复杂度。

**验证路径**：先在真实 GD32 板子上跑通 OpenOCD 连接 → 编译 → 烧录 → 串口观察，再考虑完整架构。
