# 多 Agent 工作流程

## 概述

本项目支持多 Agent 分工协作，将复杂任务分解为三个角色：

| 角色 | 职责 | 允许操作 | 禁止操作 |
|------|------|----------|----------|
| **Scout** | 收集证据和约束 | 搜索、分析、报告 | 修改代码、写入文件 |
| **Builder** | 实现代码并验证 | 编写代码、编译、烧录 | 做侦察、做审查 |
| **Verifier** | 审查和验收 | 审查、评估、报告 | 修改代码、补充实现 |

---

## 适用场景

满足以下任一条件时，启用多 Agent 模式：

- 需要修改 2 个及以上文件
- 预计需要 2 轮以上编译/烧录/调试
- 任务容易发散或需要中途回退
- 用户明确要求"多 Agent"或"并行探索"

---

## 工作流程

### 标准流程

```
用户需求
    ↓
[Scout] 收集证据和约束
    ↓
[Builder] 实现代码并验证
    ↓
[Verifier] 审查和验收
    ↓
用户确认
```

### 轮次制流程

```
Round 1:
  [Scout] 侦察 → [Builder] 实现 → [Verifier] 审查

Round 2:
  [Scout] 补充侦察 → [Builder] 修复 → [Verifier] 验收

Round N:
  直到所有功能完成并通过审查
```

## 角色职责详解

### Scout Agent

**职责**：收集证据和约束

**必须交付**：
1. 相关文件路径，按优先级排序
2. 相似实现或既有模式
3. 已确认约束：芯片、固件库、构建方式、引脚/时钟/DMA/中断限制
4. 建议的最小改动范围

**输出格式**：
```markdown
## Scout 侦察报告

### 文件定位
| 文件路径 | 用途 | 优先级 |
|----------|------|--------|

### 约束条件
- 芯片型号：xxx
- 固件库：xxx
- 引脚限制：xxx

### 候选方案
| 方案 | 来源 | 适用性 |
|------|------|--------|

### 建议
- 最小改动范围：xxx
- 下一步：xxx
```

### Builder Agent

**职责**：实现代码并验证

**必须交付**：
1. 3-7 步执行计划
2. 实际代码改动或 diff
3. 验证结果：编译/测试/烧录/日志
4. 偏差说明

**输出格式**：
```markdown
## Builder 执行报告

### 执行计划
1. xxx - 验证标准：xxx
2. xxx - 验证标准：xxx

### 代码改动
- 文件：xxx
- 改动：xxx

### 验证结果
- 编译：✅ 通过
- 烧录：✅ 通过
- 功能：✅ 正常

### 偏差说明
- 计划：xxx
- 实际：xxx
- 原因：xxx
```

### Verifier Agent

**职责**：审查和验收

**必须交付**：
1. 按严重程度排序的风险清单
2. 未覆盖的边界条件
3. 逐项审查结论
4. 最低成本补救建议

**输出格式**：
```markdown
## Verifier 审查报告

### 审查结论
✅ 通过 / ⚠️ 有条件通过 / ❌ 未通过

### 风险清单
| 严重程度 | 风险描述 | 位置 | 建议 |
|----------|----------|------|------|

### 审查详情
| 检查项 | 状态 | 说明 |
|--------|------|------|

### 补救建议
| 问题 | 建议 | 成本 |
|------|------|------|
```

## 并行保护规则

### 写入保护

- **同一时刻只允许一个 Builder 写入**
- Scout 与 Verifier 默认只读
- 需要写入时，必须获得明确授权

### 角色隔离

- Scout 不做实现决策
- Builder 不做审查结论
- Verifier 不补充实现

### 信息传递

跨角色只传：
1. 目标
2. 约束
3. 候选文件
4. 证据
5. 下一步建议

**禁止**传递整段日志或未筛选的信息。

## 轮次制管理

### 启用条件

- 修改 2 个及以上文件
- 预计需要 2 轮以上编译/烧录/调试
- 任务容易发散或需要中途回退

### 轮次规则

1. **每轮只处理一个改动点**
2. **开始前声明**：trace_id、当前轮次、目标、验证标准、停止条件
3. **结束时交付**：改动摘要、涉及文件、验证命令与输出、风险、下一轮建议

### 交接摘要格式

```text
[轮次汇报]
trace_id: <任务标识>
round: <第 N 轮>
goal: <本轮目标>
verify: <成功标准>

changes:
- <改动摘要 1>
- <改动摘要 2>

evidence:
- file: <路径:行号>
- command: <实际执行命令>
- result: <关键输出或结论>

risks:
- <已知风险或未确认项>

handoff:
1. <本轮改了什么>
2. <已确认了什么>
3. <下一轮要验证什么>
```

### 停止红线

- 新增依赖
- 一次改动超过 5 个文件
- 修改根目录构建/烧录/工具链配置
- 连续两轮都无法确认根因
- 需要危险操作（批量删除、强制回退、量产烧录）

## 证据包格式

每轮必须交付证据包：

```text
[证据包]
trace_id: <任务标识>
round: <第 N 轮>
role: <Scout/Builder/Verifier>
timestamp: <日期时间>

goal: <本轮目标>
verify: <成功标准>

changes:
- <改动摘要>

evidence:
- type: <编译/烧录/串口/寄存器>
- command: <实际执行命令>
- result: <关键输出>
- conclusion: <结论>

risks:
- <已知风险>

next: <下一步建议>
```

## 与四文件体系的集成

### 文件更新规则

| 文件 | Scout | Builder | Verifier |
|------|-------|---------|----------|
| 项目规划清单.md | 读取 | 更新进度 | 读取 |
| 编辑清单.md | 读取 | 记录修改 | 读取 |
| 硬件资源表.md | 更新约束 | 读取约束 | 核对一致性 |
| 研究发现.md | 记录搜索 | 读取方案 | 读取证据 |

### 会话恢复

会话恢复时，按以下顺序读取：
1. 项目规划清单.md → 当前阶段
2. 编辑清单.md → 最后修改
3. 硬件资源表.md → 硬件约束
4. 研究发现.md → 之前发现

## 使用示例

### 示例 1：实现 USART 串口通信

**Round 1 - Scout**：
```markdown
## Scout 侦察报告

### 文件定位
| 文件路径 | 用途 | 优先级 |
|----------|------|--------|
| src/uart.c | USART 驱动 | 高 |

### 约束条件
- 芯片：GD32F470VET6
- 固件库：标准库
- 引脚：PA9(TX)/PA10(RX)

### 建议
- 使用 USART0
- 波特率 115200
- 使用中断模式
```

**Round 1 - Builder**：
```markdown
## Builder 执行报告

### 执行计划
1. 添加 USART0 初始化函数
2. 添加中断处理函数
3. 实现 printf 重定向

### 验证结果
- 编译：✅ 通过
- 烧录：✅ 通过
- 串口：✅ 输出正常
```

**Round 1 - Verifier**：
```markdown
## Verifier 审查报告

### 审查结论
✅ 通过

### 风险清单
| 严重程度 | 风险描述 |
|----------|----------|
| 低 | 无 |

### 验收确认
- 代码质量：✅
- 硬件合规：✅
- 功能验证：✅
```

## 最佳实践

### 1. 明确角色

- 每个任务明确指定角色
- 不要混合角色职责
- 遵守角色的禁止行为

### 2. 完整交付

- 每轮必须有完整的证据包
- 证据必须可验证
- 结论必须基于证据

### 3. 及时交接

- 完成一轮后立即交接
- 交接信息简洁明了
- 避免信息过载

### 4. 遵守红线

- 遇到停止红线立即停止
- 不要强行继续
- 寻求用户确认

## 常见问题

### Q: 什么时候启用多 Agent 模式？

A: 满足以下任一条件时启用：
- 修改 2 个及以上文件
- 预计需要 2 轮以上编译/烧录/调试
- 任务容易发散或需要中途回退
- 用户明确要求

### Q: 可以跳过某个角色吗？

A: 不建议。每个角色都有其独特的价值：
- Scout 提供信息和约束
- Builder 实现和验证
- Verifier 审查和验收

### Q: 多 Agent 模式和单 Agent 模式有什么区别？

A: 
- 单 Agent：一个人完成所有工作
- 多 Agent：分工协作，各司其职

多 Agent 模式更适合复杂任务，可以提高质量和效率。
